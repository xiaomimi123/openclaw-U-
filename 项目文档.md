# OpenClaw U 盘版 — 项目文档

> 版本：v1.x 总结 | 日期：2026-03-22 | 用于重新开发参考

---

## 一、产品定位

### 我们做什么

**我们不是开发 openclaw 本身，而是为 openclaw 提供一个面向小白用户的"便携式安全封装层"。**

openclaw 本身是一个完整的 AI 网关（gateway），具备插件管理、自我更新、多渠道（微信/Telegram/Discord 等）接入能力。

**我们解决的是两个具体问题：**

| 问题 | 说明 |
|------|------|
| 部署问题 | 小白用户不会装 Node.js、不会配置环境变量、不会用命令行 |
| 数据安全 | 所有数据只存 U 盘，拔出即停止，不在任何电脑上留痕 |

### 商业模式

- **销售形式**：按 U 盘销售，一盘一授权
- **目标用户**：完全不懂技术的商用客户（微信群运营、自动回复场景为主）
- **使用场景**：插入 U 盘 → 双击 exe → 配置 API Key → 开始使用 → 拔出 U 盘即结束
- **收入来源**：U 盘硬件 + 软件授权费（一次性或订阅）
- **授权控制**：每个 U 盘绑定一个序列号，开发者用私钥签名 license.key 文件

## 二、U 盘内容结构

客户拿到的 U 盘包含以下 4 个文件/目录：

```
U 盘根目录/
├── OpenClaw-U盘版.exe        # 启动器（Electron portable，约 72 MB）
├── openclaw.zip              # openclaw 程序包（含 node_modules，约 100+ MB）
├── runtime/
│   └── node.exe              # 便携式 Node.js（约 60 MB）
└── weixin-plugin.zip         # 微信 ClawBot 插件离线包
```

运行后自动生成（不发货）：

```
U 盘根目录/
├── license.key               # 开发者颁发的授权文件（ECDSA 签名）
└── openclaw-data/            # 所有运行数据（全部存在 U 盘）
    ├── setup.json            # 初始配置记录
    ├── .openclaw/
    │   ├── openclaw.json     # openclaw 配置文件
    │   └── extensions/       # 已安装的插件
    ├── AppData/              # 重定向的 Windows AppData
    └── tmp/                  # 重定向的 TEMP 目录
```

## 三、核心技术架构

### 3.1 启动器（Electron）

**框架**：Electron 33，electron-builder portable，Windows x64

**启动流程**：
```
app.whenReady()
  → verifyLicense()          # 读取 U 盘序列号，验证 license.key ECDSA 签名
  → extractOpenclawIfNeeded() # 解压 openclaw.zip 到本机 TEMP（首次/zip 变更时）
  → createWindow()           # 显示 setup.html（首次）或 launcher.html
  → startUsbMonitor()        # 每 800ms 检测 U 盘是否存在
```

### 3.2 数据隔离（关键）

启动所有 openclaw 子进程时，注入以下环境变量，将所有数据重定向到 U 盘：

```javascript
{
  HOME:                "<USB>/openclaw-data",
  USERPROFILE:         "<USB>/openclaw-data",
  APPDATA:             "<USB>/openclaw-data/AppData/Roaming",
  LOCALAPPDATA:        "<USB>/openclaw-data/AppData/Local",
  TEMP:                "<USB>/openclaw-data/tmp",
  TMP:                 "<USB>/openclaw-data/tmp",
  npm_config_registry: "https://registry.npmmirror.com",  // 淘宝镜像
}
```

### 3.3 openclaw.zip 制作（关键，有坑）

**错误方式**：直接 `npm pack` → 缺少 node_modules → 新电脑报 `ERR_MODULE_NOT_FOUND`

**正确方式**：
```
1. git pull（openclaw 源码）
2. pnpm build（编译）
3. npm pack --ignore-scripts（生成 .tgz，跳过 prepack 重复构建）
4. npm install <tgz>（在临时目录安装，包含所有依赖的 node_modules）
5. 组装 zip-stage：openclaw 包文件放根目录 + 其他依赖放 node_modules/
6. ZipFile::CreateFromDirectory → openclaw.zip
```

**原因**：openclaw 的 `dist/` 目录有外部依赖（tslog、axios、zod、ws、yaml、express 等），没有 bundle 进去，必须附带 node_modules。

### 3.4 授权验证

```
算法：ECDSA P-256 + SHA256
私钥：开发者保管（不在代码中）
公钥：硬编码在 main.js 中
license.key 内容：sign(USB序列号) 的 hex 字符串
获取序列号：PowerShell Get-CimInstance Win32_LogicalDisk
```

颁发授权流程：
1. 客户提供 U 盘序列号（从软件"售后信息"页获取）
2. 开发者用私钥签名：`sign(serialNumber)` → hex 字符串
3. 将 hex 写入 `license.key`，放入 U 盘根目录

### 3.5 USB 监控

```javascript
// 每 800ms 检测 U 盘根目录是否可访问
// 连续 3 次失败 → 确认拔出 → 杀掉 openclaw 进程 → 2.5s 后退出 app
```

### 3.6 openclaw 子进程管理

所有 openclaw CLI 调用均通过 `spawn(node.exe, [openclaw.mjs, ...args])` 执行，共享同一套环境变量。

**已实现的功能**：

| IPC 事件 | openclaw 命令 |
|----------|--------------|
| start-openclaw | `openclaw gateway` |
| install-weixin-plugin | `openclaw plugins install <zip>` |
| update-weixin-plugin | `openclaw plugins update openclaw-weixin` |
| login-weixin-channel | `openclaw channels login --channel openclaw-weixin` |

## 四、已知问题和坑（重开发必读）

### 4.1 openclaw.zip 路径问题（最重要）

openclaw 在运行时会将插件的绝对路径写入 `openclaw.json`，例如：

```json
"plugins": {
  "entries": {
    "telegram": {
      "dir": "C:\\Users\\用户名\\AppData\\Local\\Temp\\openclaw-usb\\openclaw\\dist\\bundled\\telegram",
      "entry": "./index.js"
    }
  }
}
```

当 U 盘换到新电脑时，TEMP 路径不同（用户名不同），导致 openclaw 报：
```
Config invalid
plugins: plugin: extension entry escapes package directory: ./index.js
Run: openclaw doctor --fix
```

**根本原因**：`dist/bundled/` 里的内置插件路径被 openclaw 以绝对路径写死到了配置里，跨电脑后失效。

**推荐解法**：每次启动前扫描 `plugins.entries`，将含有 `openclaw-usb/openclaw/` 的路径更新为当前电脑的 TEMP 路径，将含有 `openclaw-data/.openclaw/extensions/` 的路径更新为当前 USB 路径。

### 4.2 openclaw build 需要 bash

`pnpm build` 的第一步是 `bash scripts/bundle-a2ui.sh`。在 Windows 上需要 WSL 或 Git bash。

解决：将 Git bash 路径（`C:\Program Files\Git\usr\bin`）插到 PATH 最前面，优先于 WSL bash。

### 4.3 npm pack 的 prepack 钩子重复构建

`npm pack` 默认会触发 `prepack` 钩子（再次 `pnpm build`）。应使用 `npm pack --ignore-scripts`，因为 step 2 已经构建过了。

### 4.4 FAT32 时间戳精度

U 盘通常是 FAT32，文件时间戳精度只有 2 秒。不能用 mtime 判断 openclaw.zip 是否变更。

**解法**：用 `size:md5(前256字节)` 作为版本指纹存在本机 TEMP。

### 4.5 MAX_PATH 限制

pnpm 的虚拟 store 路径极深，直接 Copy-Item/robocopy 复制会超过 Windows 260 字符限制。

**解法**：用 `npm install <tgz>` 在临时目录安装（npm 使用扁平 node_modules，路径短）。

### 4.6 openclaw.json 的 meta 字段

openclaw 只认识有限的 meta 字段。如果 meta 里有多余字段（如 description），会导致 Config invalid。

**解法**：每次写 openclaw.json 前清理 meta，只保留 `lastTouchedVersion` 和 `lastTouchedAt`。

## 五、UI 功能模块

### setup.html（首次配置向导）
- 选择 AI 服务商（Anthropic/OpenAI/DeepSeek/通义/GLM）
- 输入 API Key
- 选择聊天渠道（微信/Telegram/Discord/飞书/无）
- 完成后写入 `openclaw.json` 和 `setup.json`

### launcher.html（主控制台）
- **状态栏**：运行中/启动中/已停止，带动画指示灯
- **环境检查（preflight）**：Node.js / openclaw 文件 / 端口 18789 / 配置文件 / U 盘剩余空间
- **实时日志**：gateway 输出流显示
- **控制按钮**：启动 / 停止 / 打开 Web UI / 一键修复
- **API Key 管理**：切换服务商 + 更新 Key，运行中自动重启
- **微信插件区**：安装插件 / 扫码登录 / 更新插件
- **售后信息**：版本号 + U 盘序列号，供客服核验

## 六、重新开发建议

### 6.1 保留不变的部分
- 授权验证逻辑（ECDSA 签名方案成熟可靠）
- 数据隔离方案（环境变量重定向到 USB，核心设计正确）
- USB 监控逻辑（连续 3 次失败才判断拔出，防抖设计正确）
- openclaw.zip 制作流程（`npm install <tgz>` 方式）

### 6.2 重点改进方向

**问题 1：插件路径跨电脑失效**

openclaw 把内置插件的绝对路径写入配置，这是 openclaw 本身的设计。
建议：启动器在写 openclaw.json 时，**不要在 `plugins.entries` 里写绝对路径**，或者每次启动前用正则将含有 TEMP/USB 路径的条目更新为当前路径。

更好的方案：保持 `plugins.entries` 为空 `{}`，让 openclaw 自己从 `dist/bundled/` 里发现内置插件，只记录用户安装的插件（weixin 等），且使用相对路径。

**问题 2：openclaw.json 格式随版本变化**

openclaw 每次升级可能改变配置 schema，导致旧配置 invalid。
建议：维护一个"最小可启动配置"模板，每次升级后验证配置是否仍然合法。

**问题 3：对 openclaw 版本的强依赖**

建议：锁定 openclaw 版本（不自动 git pull），只在测试通过后手动升级。

### 6.3 技术栈建议（维持现有）

```
启动器:  Electron 33 + electron-builder portable
打包:    Windows x64 portable exe（单文件，无需安装）
UI:      纯 HTML/CSS/JS（无框架，加载快，无 node_modules 依赖）
Node.js: 便携版 node.exe 随 U 盘发货
授权:    ECDSA P-256，Node.js crypto 内置，无外部依赖
```

### 6.4 openclaw 配置最小模板

```json
{
  "meta": { "lastTouchedVersion": "1.x.x", "lastTouchedAt": "..." },
  "gateway": {
    "mode": "local",
    "auth": { "mode": "token", "token": "<随机 48 字节 hex>" },
    "controlUi": { "allowInsecureAuth": true, "dangerouslyDisableDeviceAuth": true }
  },
  "env": { "<PROVIDER>_API_KEY": "<用户 key>" },
  "agents": { "defaults": { "model": { "primary": "<provider/model>" } } },
  "plugins": { "entries": {} },
  "update": { "checkOnStart": false }
}
```

**关键原则**：`plugins.entries` 保持空 `{}`，不要写入任何路径信息，让 openclaw 自己管理插件发现。

## 八、发货清单与流程

### 发货文件
| 文件 | 说明 |
|------|------|
| `OpenClaw-U盘版.exe` | `npm run build` 生成，放 U 盘根目录 |
| `openclaw.zip` | 运行 `同步U盘版.ps1` 生成，放 U 盘根目录 |
| `runtime\node.exe` | 便携 Node.js，放 U 盘 runtime/ 目录 |
| `weixin-plugin.zip` | 微信插件离线包，放 U 盘根目录 |
| `license.key` | 用私钥签名客户 U 盘序列号，单独颁发 |

### 颁发授权
1. 客户安装后打开软件 → 售后信息页 → 获取 U 盘序列号
2. 开发者运行签名脚本：`sign(serialNumber)` → 生成 `license.key`
3. 将 `license.key` 发给客户，放入 U 盘根目录
4. 重启软件即可激活
